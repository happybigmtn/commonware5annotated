# Chapter 05 — P2P: authenticated, encrypted peer channels

> How Commonware ships signed messages to specific peers over a real network.

## The problem

Chapter 04 gave us signatures. Chapter 01 gave us consensus messages that need
to be **shipped** between specific peers. Now we need the network layer.

Three things must be true:

1. **Authenticated.** When a message arrives claiming to be from peer X, the
   recipient can cryptographically verify it's really X. Not just "X's IP
   address" — actual cryptographic proof of identity.
2. **Encrypted.** Nobody between us and X can read the message. Not the
   network operator, not a man-in-the-middle.
3. **Routed.** Different message types go to different handlers (channels).
   The consensus vote handler doesn't want to receive gossip messages.

Plus operational concerns: rate limiting (don't let X spam you), blocking
(don't talk to X anymore if X is misbehaving), peer set management (who are
my peers anyway?), connection lifecycle (dial, accept, keep alive,
disconnect).

Commonware ships all of this in `p2p/`.

## The trait surface

Open `p2p/src/lib.rs`. The traits you'd use as a caller:

```rust
pub type Message<P> = (P, IoBuf);                 // (sender_pk, payload_bytes)

pub enum Recipients<P: PublicKey> {              // who to send to
    All,                                         // everyone in the active peer set
    Some(Vec<P>),                                // specific list
    One(P),                                      // a single peer
}

pub trait UnlimitedSender {                       // send without rate limiting
    fn send(&mut self, recipients: Recipients<P>,
            message: impl Into<IoBufs>, priority: bool) -> Unreliable<Feedback>;
}

pub trait LimitedSender: UnlimitedSender {        // send with rate limiting
    fn check(&mut self, recipients: Recipients<P>) -> Result<CheckedSender, ...>;
}

pub trait Receiver {                              // receive from anyone
    async fn recv(&mut self) -> Result<Message<P>, Error>;
}

pub trait Manager {                               // peer set management
    fn track(&mut self, id: u64, peers: TrackedPeers<P>) -> Feedback;
}

pub trait Blocker {                               // cut off a peer
    fn block(&mut self, peer: P) -> Feedback;
}
```

The `Sender` trait is the interesting one. It checks rate limits first,
filters out peers who exceeded their quota, then sends to the rest. If all
peers are rate-limited, you get an error with the time when the earliest one
will be available again.

```rust
fn check(&mut self, recipients: Recipients<P>) -> Result<CheckedSender, SystemTime>
```

Notice the error type is `SystemTime`. The rate limiter is telling you:
"everyone you wanted is throttled; try again at this instant." That's not an
error in the usual sense — it's a backpressure signal. The caller can choose
to wait, drop the message, or pick a different set of recipients.

## The `Data` envelope and the `Payload` enum

`p2p/src/authenticated/data.rs:7-39`:

```rust
pub struct Data {
    pub channel: u64,         // routes the message to a handler
    pub message: IoBuf,       // arbitrary bytes (your consensus message, encoded)
}
```

Every "real" message is wrapped in a `Data` with a `channel` ID. The
consensus engine registers channel 0 for votes, channel 1 for certificates,
channel 2 for resolver requests (from chapter 01's simplex config). When a
message arrives, the network layer looks at `channel` and routes it to the
right mailbox.

`p2p/src/authenticated/discovery/types.rs:62-82`:

```rust
pub enum Payload<C: PublicKey> {
    Data(Data),                // the actual application message
    Greeting(Info<C>),         // MUST be the first message on a new connection
    BitVec(BitVec),            // "I know about these peers" — also serves as ping
    Peers(Vec<Info<C>>),       // gossip: here are some peer addresses
}
```

Four message types, four purposes:

1. **`Greeting`** — sent once at the start of every connection. Contains
   your public key, your reachable addresses, your signature. If you don't
   send this first, the connection is killed. If you send it twice, killed.
   It's the handshake.

2. **`Data`** — the actual application payload, channel-tagged.

3. **`BitVec`** — a compact bitmap saying "I know these peers exist." This
   also serves as a keep-alive ping when there's no real traffic.

4. **`Peers`** — gossip. "Here are some peer addresses I learned about."
   Lets discovery mode learn the network without bootstrap.

Each is `1`-byte prefix (`0`, `1`, `2`, `3`) + payload, so the reader knows
what it's decoding.

## Two connection modes: lookup vs discovery

`p2p/src/authenticated/mod.rs:1-15`:

```
[discovery] operates under the assumption that peer addresses aren't known in
advance, and that they need to be discovered. Bootstrappers are used to
connect to the network and discover peers.

[lookup] operates under the assumption that peer addresses are known in
advance, and that they can be looked up by their identifiers.
```

| Mode | When to use | Bootstrapping |
|---|---|---|
| `lookup` | Static validator set, addresses pre-configured | None needed |
| `discovery` | Dynamic networks, addresses learned at runtime | A few "bootstrap" peers |

In **lookup mode** (the `lookup` submodule), you give the network a list of
`(public_key, socket_addr)` tuples. It dials them, handshakes, done. No
gossip needed.

In **discovery mode** (the `discovery` submodule), you give it **bootstrappers**
— a small initial set of peers. It dials them, sends its `Greeting`, and then
exchanges `Peers` messages to learn about more peers. The network grows.

## The handshake — how you prove "I'm really pk X"

Both modes use the same handshake, built on `commonware-stream::encrypted`
(which uses ChaCha20-Poly1305 for symmetric crypto). The flow:

1. Dialer opens a TCP connection to the listener.
2. Listener accepts.
3. Dialer sends its `Greeting` (signed by its private key).
4. Listener sends its `Greeting`.
5. Both sides verify the other's signature.
6. Both sides derive a shared session key via X25519 Diffie-Hellman.
7. All subsequent traffic on this connection is encrypted with that key.

If any step fails (wrong signature, expired greeting, stale timestamp,
handshake timeout), the connection is killed.

Look at `config.rs:65-92` — the knobs:

```rust
pub synchrony_bound: Duration,                  // how far in the future a timestamp can be
pub max_handshake_age: Duration,                // greeting must be this recent
pub handshake_timeout: Duration,                // total time to complete handshake
pub max_concurrent_handshakes: NonZeroU32,      // cap on in-flight handshakes
pub allowed_handshake_rate_per_ip: Quota,       // per-IP rate limit
pub allowed_handshake_rate_per_subnet: Quota,   // per-subnet rate limit
```

These exist because **handshake = unauthenticated work**. Before you've
verified the peer's signature, the peer could be anyone. So you bound the
work: short timeout, low rate limits. After the handshake, you're
authenticated and the connection can carry lots of traffic.

The `synchrony_bound` is for **replay protection** — if the greeting claims
to be signed at time T, but T is 10 minutes in the future, reject. (A real
clock is loosely synchronized; an attacker can drift arbitrarily.)

## Rate limiting per channel

When you register a channel, you specify a `Quota`:

```rust
let (vote_sender, vote_receiver) = network.register(
    0,                          // channel ID
    Quota::per_second(NZU32!(10)),  // 10 messages/sec/peer
    256,                        // 256 messages in flight
);
```

Each peer on this channel can send you at most 10 messages per second. If
they exceed it, the `LimitedSender::check` filters them out — your message
goes to everyone else, just not them.

This is **defense in depth**. Your consensus engine can already handle
malformed messages (the `Blocker` trait cuts off attackers). Rate limiting
just means an attacker can't burn your CPU at line rate before you have a
chance to notice.

## The Blocker — cutting off misbehaving peers

`p2p/src/lib.rs:362-368`:

```rust
pub trait Blocker: Clone + Send + Sync + 'static {
    fn block(&mut self, peer: P) -> Feedback;
}
```

When you detect that peer X is sending you invalid signatures, malformed
messages, or anything else that suggests malice, call `block(X)`. The
network layer:

1. Severs any existing connection to X.
2. Refuses all future connections from X.
3. Logs the event.

The `block!` macro (`p2p/src/lib.rs:391-398`) wraps this in a `tracing::warn!`:

```rust
macro_rules! block {
    ($blocker:expr, $peer:expr, $($arg:tt)+) => {
        let peer = $peer;
        tracing::warn!(peer = ?peer, $($arg)+);
        $blocker.block(peer)
    };
}
```

Use it everywhere you would have just called `.block()`. Now every block
site has a structured log entry explaining why.

## Tracked peer sets and the active set

`Manager::track(id, peers)` registers a peer set with a monotonically
increasing `id`. The most recent ID is the **active set** — the network
maintains connections to those peers and disconnects from the rest.

But it doesn't fully drop connections to older peer sets. `tracked_peer_sets`
in the config tells the network how many historical sets to keep alive.
**Why?** Two reasons:

1. **Graceful eviction.** When validator X is removed from the active set,
   it might still need to download missing blocks. Keeping the connection
   open for a few minutes helps.
2. **Resharing.** When the threshold key is being reshared (chapter 04), the
   old and new committees both need to participate briefly. Tracking
   multiple sets lets them coexist.

`TrackedPeers { primary, secondary }` (`p2p/src/lib.rs:205-212`):

> **Primary** peers help "drive progress" — they participate in consensus.
> **Secondary** peers "follow that progress" — they receive but don't
> contribute. Primary get more bandwidth and lower latency.

This is the same pattern as "full nodes" vs "light clients" in other
systems. Commonware just makes it a first-class concept.

## How messages actually move

Pulling it all together — when consensus sends a `notarize` message:

```
Simplex engine
    │
    │  sender.send(Recipients::All, encoded_notarize_msg, false)
    ▼
LimitedSender::check                              ◄─── rate-limit filter
    │
    │  CheckedSender (only non-throttled peers)
    ▼
CheckedSender::send                               ◄─── per-peer send
    │
    │  For each peer:
    ▼
authenticated::data::Data { channel: 0, message }
    │
    │  Wrap in Payload::Data, encrypt with per-peer session key
    ▼
commonware-stream::encrypted::Sender
    │
    │  TCP write to peer
    ▼
peer receives, decrypts, sees Payload::Data, routes to channel 0 mailbox
```

The whole stack runs on top of the abstract `Network` trait from chapter 02.
In production, that's Tokio + real TCP. In tests, it's the simulated network
from chapter 02 with all its bandwidth simulation, partitions, etc.

## The OSI model — where P2P actually lives

Before diving deeper, it's worth knowing **where Commonware's P2P sits in
the network stack**. The 7-layer OSI model is mostly obsolete in practice,
but it's still a useful vocabulary.

```
+---------------------------+
| 7: Application            |  <-- Commonware's p2p/, consensus, runtime
+---------------------------+
| 6: Presentation           |  <-- (empty in Commonware)
+---------------------------+
| 5: Session                |  <-- (empty in Commonware)
+---------------------------+
| 4: Transport              |  <-- TCP (reliable bytes) / UDP (datagrams)
+---------------------------+
| 3: Network                |  <-- IP (routing)
+---------------------------+
| 2: Data Link              |  <-- Ethernet, Wi-Fi
+---------------------------+
| 1: Physical               |  <-- Fiber, copper, radio
+---------------------------+
```

Commonware's P2P is **layer 7** — application layer. It runs on top of:

- **Layer 4:** TCP (via Tokio or the runtime abstraction). Provides
  reliable, ordered, byte-stream transport.
- **Layer 3:** IP. Routes packets between hosts.
- **Layer 2/1:** Whatever the host's NIC provides.

Commonware does **not** implement routing (layer 3) or transport (layer 4).
It assumes TCP is available. The reason: implementing transport
correctly is enormously hard (congestion control, NAT traversal,
retransmission, head-of-line blocking). Better to use the OS's TCP.

What Commonware adds on top of TCP:

- **Encryption** (so the network operator can't read messages).
- **Authentication** (so a peer knows it's really talking to validator X).
- **Routing** (so consensus votes go to the vote handler, gossip to the
  gossip handler).
- **Rate limiting** (so peer X can't burn peer Y's CPU).
- **Block lists** (so peer Y can cut off peer X).

All of these are **layer 7 concerns**. Encryption and authentication
could arguably be layer 5 or 6, but Commonware bundles them into the
application layer for simplicity.

The encapsulation story:

```
Commonware message (Rust struct)
  -> codec::encode (chapter 03) -> bytes
  -> stream::encrypted (X25519 + ChaCha20-Poly1305) -> ciphertext
  -> TCP socket -> IP packet -> Ethernet frame
```

At the receiving side, the same layers unwrap in reverse. Each layer
only knows about the layer directly above and below it.

## TCP vs UDP — why Commonware chose TCP

Two transport options for application protocols: TCP and UDP.

### TCP — reliable byte stream

- **Guarantees:** in-order, no duplicates, no loss. The OS handles
  retransmission, ordering, and flow control.
- **Cost:** head-of-line blocking (a lost segment stalls all later
  segments until it's retransmitted), connection setup overhead (3-way
  handshake), congestion control (sender slows down when network is
  congested).
- **Use case:** when you need **reliability** more than **latency**.

### UDP — best-effort datagrams

- **Guarantees:** none. Packets can be lost, reordered, duplicated.
- **Cost:** none beyond the per-packet header. Low latency, no setup.
- **Use case:** when you need **latency** more than **reliability**.
  Real-time voice, video, gaming, DNS queries.

### Commonware's choice

`commonware-stream::encrypted::Sender` uses TCP (chapter 05 main text
and Appendix A). Why?

BFT consensus has an **implicit reliability requirement**: if a vote
is lost, the protocol stalls. The whole point of a 2-round or 3-round
consensus protocol is that messages arrive reliably. If 30% of votes
are lost in transit, you can't reach a quorum.

UDP would require reimplementing reliability on top: retransmission
timers, sequence numbers, deduplication. That's RUDP, and it's a lot
of code. Better to use TCP and let the OS handle it.

The cost — head-of-line blocking — is acceptable because consensus
messages are small (typically under 10 KB) and the network is well-
behaved enough that retransmissions are rare.

### What about the `nullify` on timeout?

If a peer times out (chapter 01), the protocol sends a `nullify` and
moves to the next view. This is the consensus layer's response to TCP
failures. It says: "I never saw the vote, so I'll nullify the view and
let a new leader try." This is **the application compensating for
transport unreliability** — the timeout gives the OS enough retries
before declaring failure.

If Commonware used UDP, you'd see the same `nullify` pattern but for
different reasons (packet loss rather than TCP backpressure). The
consensus protocol doesn't care which transport failed; it just knows
"no quorum in time, nullify."

### When UDP would be right

For protocols that are latency-critical and can tolerate some loss,
UDP is correct. Examples: real-time streams (live video), high-
frequency trading (where stale prices are useless), multiplayer games
(where lost state can be reconstructed client-side).

For BFT consensus, TCP wins. The message rate is moderate (not
millions per second), the reliability requirement is strict, and the
latency cost of retransmission is dwarfed by the latency of waiting for
the next view.

## Authenticated encryption — the AEAD concept

Commonware's transport uses **authenticated encryption with associated
data** (AEAD). The construction: ChaCha20-Poly1305. The concept
deserves a section.

### The problem AEAD solves

Plain encryption provides **confidentiality** (the attacker can't read
the message) but not **integrity** (the attacker can flip bits, and
the receiver might not notice). Worse, in a network protocol, the
attacker can rearrange messages, replay old messages, or substitute
different messages.

For BFT, every one of these attacks is fatal. A flipped bit in a vote
could change "vote for block B" to "vote for block B'." A replayed old
vote could appear to be a fresh one. A substituted message could look
like it came from a different validator.

**AEAD** provides all of:
1. **Confidentiality** (encrypted content).
2. **Integrity** (any tampering is detected).
3. **Authenticity** (the receiver knows who sent it).
4. **Replay resistance** (each nonce is unique; old messages can't be
   re-submitted).

### ChaCha20-Poly1305

The cipher: ChaCha20 (stream cipher). The MAC: Poly1305. Together, an
AEAD that:

- Encrypts the plaintext with ChaCha20 using a key and nonce.
- Computes a Poly1305 tag over `(ciphertext || associated_data)`.
- Outputs `(ciphertext, tag)`.

The receiver:
- Decrypts with ChaCha20 using the same key and nonce.
- Recomputes the Poly1305 tag over `(ciphertext || associated_data)`.
- Compares in constant time (`subtle::ct_eq`).
- If the tags match, the message is authentic and unmodified.

### The "associated data" is the peer identity

This is the trick that makes Commonware's P2P **identity-bound encryption**.

```
AEAD_encrypt(
    key,             // session key from X25519
    nonce,           // counter
    associated_data, // peer's public key
    plaintext,
) -> (ciphertext, tag)
```

The `associated_data` is NOT encrypted. It's authenticated, meaning:
the receiver checks that the message was encrypted with the **expected
sender's public key** baked in. If an attacker captures a ciphertext
from peer A to peer B, and tries to replay it to peer C, peer C will
see that the associated data says "peer A" but the connection is
supposed to be with peer A's identity already pinned.

Actually, more precisely: the associated data is whatever the protocol
binds to the message. Commonware uses the **peer identity** (long-term
public key) so that an attacker can't replay a message from one
connection to another even if both connections use the same session
key.

### ChaCha20-Poly1305 vs AES-GCM

Both are standard AEADs. Trade-offs:

| Property | ChaCha20-Poly1305 | AES-GCM |
|---|---|---|
| Hardware acceleration | ARM (NEON), some x86 (AVX-512) | x86 (AES-NI), ARM (some) |
| Software fallback speed | Fast everywhere | Slow without hardware |
| Key size | 256 bits | 128 or 256 bits |
| Nonce size | 96 bits (recommended) | 96 bits (recommended) |
| Security margin | Generous (20+ rounds) | Tighter (some recent attacks on reduced-round AES) |
| Patent encumbrance | None | None (expired) |

Commonware picks **ChaCha20-Poly1305** because:

1. **Software fallback is fast.** A CPU without hardware AES still
   encrypts at full speed.
2. **No timing side-channels on key schedule.** ChaCha20 is ARX-based
   (add-rotate-xor); AES's key schedule is harder to make constant-time.
3. **Generous security margin.** 20 rounds of ChaCha20 vs 10-14 of AES.
4. **Cloud-friendly.** Cloud VMs often have AES-NI, but edge devices
   (phones, embedded validators) may not.

The `blst` library doesn't matter here (BLS is signing, not
encryption). For symmetric crypto, ChaCha20-Poly1305 is the choice.

## The X25519 + ChaCha20-Poly1305 handshake

The full handshake is implemented in `commonware-stream/src/encrypted/handshake.rs`.
Let me walk through it conceptually.

### The setup: long-term keys and ephemeral keys

Both peers have a **long-term key pair** (`sk_long`, `pk_long`). In
Commonware's case, this is an Ed25519 signing key — used for the
signed `Greeting` (chapter 05 main text).

For the key exchange, they also generate **ephemeral** X25519 keys
(`sk_eph`, `pk_eph`) at handshake time. Ephemeral keys are destroyed
after the handshake completes.

### The handshake flow

```
Dialer                                                  Listener
  |                                                        |
  |--- TCP SYN --->                                        |
  |<--- TCP SYN-ACK ---                                    |
  |--- TCP ACK --->                                        |
  |                                                        |
  |--- ephemeral_pk_d, encrypted_payload_d --->            |
  |     (encrypted_payload_d contains: signed Greeting    |
  |      with sk_long_d, signed timestamp)                 |
  |                                                        |
  |<--- ephemeral_pk_l, encrypted_payload_l ---            |
  |     (encrypted_payload_l contains: signed Greeting    |
  |      with sk_long_l, signed timestamp)                 |
  |                                                        |
  | Both sides compute:                                    |
  |   shared_secret = X25519(sk_eph, pk_eph_peer)         |
  |   session_key = HKDF(shared_secret, role||namespace)   |
  |                                                        |
  | Both sides verify the other's Greeting signature      |
  |   against the expected peer set.                       |
  |                                                        |
  | All subsequent traffic: ChaCha20-Poly1305(session_key) |
```

### Why X25519

X25519 is a Diffie-Hellman key exchange on Curve25519 (the Montgomery
form of the curve Ed25519 uses in Edwards form). Properties:

- **Fast.** ~1 microsecond per operation on modern hardware.
- **Constant-time.** The Montgomery ladder implementation resists timing
  attacks.
- **Small.** 32-byte public keys, 32-byte shared secrets.
- **No configuration.** Unlike RSA, there's no choice of key size, no
  padding mode, no exponent. Use it once.

Why not RSA for key exchange? RSA key exchange has the same security
properties as X25519 (against classical attackers), but:

- RSA is much larger (2048+ bits vs 256 bits).
- RSA has a worse track record of implementation bugs (Bleichenbacher
  attacks on PKCS#1 padding).
- RSA doesn't give you forward secrecy by default (you can do ephemeral
  RSA, but it's slower and rarer).

### Perfect forward secrecy (PFS)

If an attacker records all encrypted traffic today, and 10 years from now
they steal your **long-term private key**, can they decrypt the recorded
traffic?

With **RSA** (no PFS): yes. They use your long-term key to decrypt the
session keys.

With **X25519 + ephemeral keys** (PFS): no. The session keys are derived
from ephemeral keys, which are destroyed after the handshake. The
long-term key only authenticates the handshake, but it doesn't reveal
the session key.

This is what ephemeral keys buy you. The Commonware handshake uses them.

### The handshake payload

The "encrypted_payload" inside the handshake is encrypted with the
**session key** derived from the X25519 exchange. So even if an
attacker sees the wire, they can't read the Greeting until after the
key exchange succeeds.

The Greeting itself contains:

```rust
pub struct Info<C: PublicKey> {
    pub public_key: C,           // long-term identity
    pub addresses: Vec<Ingress>, // reachable sockets
    pub timestamp: SystemTime,   // for replay protection
    pub signature: Signature,    // over (pk || addresses || timestamp)
}
```

The signature is over the public key + addresses + timestamp. The
listener verifies: (a) the signature is valid, (b) the timestamp is
within `max_handshake_age` of now, (c) the public key is in the
expected peer set.

If any of these fail, the connection is killed. Handshake aborts are
silent from the perspective of subsequent traffic — the listener
discards the connection without further handshake messages.

## Rate limiting math — the governor crate

`p2p` uses the `governor` crate for per-peer rate limits. Understanding
how it works helps you tune the quotas.

### The token bucket algorithm

The classic algorithm. Each peer has a **bucket** that holds up to `B`
**tokens**. Tokens are added at rate `R` per second. To send a message,
the sender spends 1 token. If the bucket is empty, the message is
rejected (or queued, depending on policy).

```
bucket state: tokens = B (initially full)

on send:
    if tokens > 0:
        tokens -= 1
        send message
    else:
        reject message (or backpressure)

every 1/R seconds:
    tokens = min(B, tokens + 1)
```

Properties:

- **Burst capacity: B.** A peer can send up to B messages in a tight
  burst before getting throttled.
- **Steady rate: R.** Over time, the peer can sustain R messages per
  second.
- **Smoothing.** The bucket smooths out bursts; you can't permanently
  exceed R.

### The governor crate

`governor::RateLimiter` implements token bucket with these properties:

- **Per-key buckets.** Each peer gets its own bucket.
- **Atomic state.** The bucket is stored in an `Arc<...>` so multiple
  threads can check it concurrently.
- **Lazy refill.** Instead of a background timer, the bucket refills
  "just in time" when checked. This avoids a global timer.
- **Direct & keyed variants.** `DirectRateLimiter` for global rate
  limits, `DefaultKeyedRateLimiter` for per-key.

In Commonware:

```rust
type Quota = NonZeroU32;
let q = Quota::per_second(NZU32!(10));   // 10/sec, burst 10
let limiter: DefaultKeyedRateLimiter<Pk> = RateLimiter::keyed(q);
```

When you call `limiter.check_key(&peer_pk)`, governor returns either
`Ok(())` (peer can send) or `Err(NotUntil<...>)` (peer is throttled;
the `NotUntil` carries the time when the bucket will refill).

### Why per-peer AND per-channel

Without per-channel limits, a peer could fill one channel's quota and
starve others. Imagine peer X sending 10 messages/sec on channel 0 (vote)
and you need to receive 10 messages/sec on channel 1 (resolver).
Channel 0 fills its quota; channel 1 is starved.

With per-channel limits, each channel has its own quota per peer. Peer X
gets 10/sec on channel 0 AND 10/sec on channel 1 independently.

`Commonware` enforces this by maintaining a separate `RateLimiter` per
`(peer, channel)` pair.

### The backpressure signal

`LimitedSender::check` returns `Result<CheckedSender, SystemTime>`. The
error type is **deliberately not an `Error`**. It's a `SystemTime`
indicating when the earliest-throttled peer will be available.

This means the caller can:

1. **Wait** until the indicated time, then retry.
2. **Drop** the message and move on.
3. **Pick different recipients** if available.

The `CheckedSender` returned on success is bound to a specific set of
peers (those who passed the rate check). The caller can't accidentally
send to a throttled peer later — the binding prevents it.

## NAT traversal — STUN, TURN, ICE, and the bootstrapper pattern

Most validators aren't directly addressable. They're behind NAT
(Network Address Translation), which means:

- They have a private IP (e.g., `192.168.1.42`).
- The router maps outbound connections to a public IP:port.
- Inbound connections from the internet hit the router, which doesn't
  know which internal host to send them to.

For BFT, this is a problem: if validator X is behind NAT, no one else
can dial them directly. But X can dial out. So the question becomes:
how do peers connect to X?

### The three NAT-traversal technologies

**STUN** (Session Traversal Utilities for NAT): a server that tells a
peer what its public IP appears as. Doesn't actually open any ports;
just maps private → public. Works for some NAT types ("full cone"),
fails for others ("symmetric").

**TURN** (Traversal Using Relays around NAT): a relay server. The
NAT-ed peer sends all traffic to TURN, TURN forwards. Works always,
but doubles the bandwidth and adds latency.

**ICE** (Interactive Connectivity Establishment): combines STUN and
TURN. Each peer tries multiple connection paths (direct, STUN-derived,
TURN-relayed), tests each, picks the best.

These are the WebRTC stack. They're complex.

### Commonware's approach: bootstrapper

`p2p/src/authenticated/discovery/` uses **bootstrappers**, not STUN/TURN.
The flow:

1. Operator configures a small set of **bootstrappers** — publicly
   reachable peers that everyone can connect to.
2. Each validator dials a bootstrapper.
3. Validators exchange peer addresses via the `Peers` gossip message
   (chapter 05 main text).
4. Each validator now has addresses for many other validators, some of
   which may be dialable directly (depending on NAT).

The bootstrapper pattern is simpler than ICE because it doesn't try to
punch through NATs. It assumes **some peers are publicly reachable**
(e.g., cloud-hosted validators with public IPs) and uses them as
**rendezvous points**.

### Why this is right for BFT

BFT consensus doesn't require every peer to be reachable. It requires
**enough** peers to be reachable to maintain quorum. If 60% of
validators are cloud-hosted (public IPs) and 40% are behind NAT, the
40% can still participate — they just can't be dialed directly.

The bootstrapper approach also dodges the WebRTC-style complexity.
There's no STUN server, no TURN relay, no ICE candidate gathering.
There's just: "dial the configured bootstrappers, learn more addresses,
dial them."

The cost: you need a small set of always-on, publicly-reachable
bootstrappers. For a permissioned BFT network (e.g., a 100-validator
set), the operator can designate 5-10 cloud-hosted validators as
bootstrappers.

## Connection state machine — the full picture

Appendix C of the original chapter showed a 5-state machine. Let me
expand it with the **all** transitions and the role of `disconnect_on_block`.

```
        ┌─────────┐
        │  Init   │
        └────┬────┘
             │ accept() / dial()
             ▼
        ┌─────────┐
        │ Wait    │   (waiting for peer's Greeting)
        │ Greet   │
        └────┬────┘
             │ recv(Greeting) - first message
             ▼
        ┌─────────┐
        │ Send    │   (send our Greeting)
        │ Greet   │
        └────┬────┘
             │ send(Greeting)
             ▼
        ┌──────────────────────┐
        │ Active               │   (encrypted channel ready)
        │ - send/recv Data     │
        │ - send/recv BitVec   │
        │ - send/recv Peers    │
        └────┬─────────────────┘
             │ any of:
             │ - peer disconnects
             │ - handshake timeout
             │ - idle timeout
             │ - protocol error (bad sig, malformed)
             │ - local close (during shutdown)
             ▼
        ┌─────────┐
        │  Dead   │   (cleanup, no more messages)
        └─────────┘
```

The transitions are **driven by the connection task**, which reads from
the socket and writes to the channel's mailbox (chapter 15).

### The `disconnect_on_block` flag

In the discovery config (`p2p/src/authenticated/discovery/config.rs`),
there's a `disconnect_on_block: bool` field. What does it do?

When `disconnect_on_block = true`: as soon as the application calls
`blocker.block(peer)`, the network layer immediately tears down all
connections to that peer. The peer is **forcibly disconnected**.

When `false`: existing connections to a blocked peer stay open until
they naturally close (idle timeout, peer disconnect, etc.), but no new
connections are accepted.

For BFT, `disconnect_on_block = true` is typical: if peer X is sending
malformed messages, you want them off your network **now**, not "when
their current TCP segment expires."

### Per-task responsibilities

The connection state machine is split across tasks:

- **Dialer (spawner):** owns outbound connections. Looks up which peers
  in the active set aren't yet connected, dials them.
- **Listener (ingress):** owns inbound connections. Accepts from the TCP
  listener socket, does the handshake.
- **Connection task:** one per active connection. Reads frames from the
  socket, writes to the channel mailbox. Handles frame parsing,
  encryption, decryption.
- **Shutdown coordinator:** signals all tasks to stop on graceful
  shutdown.

The tasks communicate via Tokio channels or the runtime's `mpsc`
abstraction (chapter 02). On error, a task sends a signal to the
coordinator, which decides whether to restart, block the peer, or
initiate shutdown.

## Peer discovery — three models

Commonware's P2P supports two of the three common discovery models. Let
me walk through all three.

### Model 1: Static / lookup

You have a pre-configured list of `(public_key, socket_addr)` tuples.
You dial them directly. No gossip.

Pros: simplest, fully predictable, no bootstrap dependency.

Cons: doesn't scale to dynamic peer sets, requires out-of-band
configuration every time the set changes.

Commonware's `p2p/src/authenticated/lookup/` implements this. Used for
permissioned BFT networks where the validator set is known in advance.

### Model 2: DNS-based

You publish peer addresses in DNS records (`_commonware._tcp.example.com`).
Peers query DNS to discover. Optional DNS seed rotation for privacy.

Pros: simple, no gossip, semi-dynamic.

Cons: DNS is a single point of failure, requires DNS infrastructure,
some censors intercept DNS.

Used by Bitcoin (seed nodes in DNS), some Ethereum clients. Commonware
doesn't ship DNS discovery, but you could implement it on top of the
abstract `Manager` trait.

### Model 3: Gossip-based

Peers exchange "I know about these other peers" messages. Over time,
each peer learns about more peers.

Pros: fully dynamic, no central server.

Cons: bootstrap problem (you need at least one initial peer), gossip
overhead, potential for pollution (a malicious peer can advertise fake
addresses).

Commonware's `p2p/src/authenticated/discovery/` implements this. Used
for dynamic networks, public chains.

### Bootstrap-strapping

In gossip mode, the bootstrap problem is solved by **bootstrappers**:
a small set of well-known, publicly-reachable peers. Each new peer
dials a bootstrapper first, exchanges `Peers` messages, and learns the
rest of the network.

Bootstrap addresses are like DNS seeds: they're a starting point, not
the final word. Once a peer knows enough other peers, it can lose
contact with bootstrappers and still find more peers via gossip.

## Rust pattern — `Arc<RwLock<...>>` vs `Mutex` vs atomics

Commonware's P2P uses different synchronization primitives for different
data. Knowing why helps you read the code.

### `Arc<T>` for read-mostly shared state

Some data is read often but written rarely. Like the active peer set:

```rust
struct NetworkState {
    active_peers: Arc<RwLock<HashSet<PublicKey>>>,
    ...
}
```

The `Arc` lets multiple tasks hold a reference to the same set. The
`RwLock` allows multiple concurrent reads but exclusive writes.

For a peer set that changes occasionally (e.g., on epoch transition)
but is read on every message send, `RwLock` is the right choice.

### `Arc<Mutex<...>>` for write-heavy or short critical sections

Some data is updated on every event. Like per-peer rate-limit state:

```rust
struct PeerState {
    rate_limiters: Arc<Mutex<HashMap<PublicKey, RateLimiter>>>,
    ...
}
```

A `Mutex` serializes access; only one task can be in the critical
section at a time. Use `Mutex` when:

- The critical section is short.
- Writes are frequent.
- You can't easily use `RwLock` (e.g., the protected state is itself
  not `Sync`).

Commonware uses `parking_lot::Mutex` rather than `std::sync::Mutex` for
the standard reasons:

- `parking_lot::Mutex` is faster (smaller, fewer atomic ops on the
  uncontended path).
- `parking_lot::Mutex` doesn't poison on panic.
- `parking_lot::Mutex` has a `try_lock_for(duration)` method.

### Atomics for simple counters

For "increment this counter" or "set this flag," atomic operations are
best:

```rust
struct Stats {
    messages_sent: AtomicU64,
    bytes_sent: AtomicU64,
    active_connections: AtomicUsize,
}
```

`AtomicU64::fetch_add(1, Ordering::Relaxed)` is lock-free, scales to
many threads, and is faster than `Mutex<u64>` for simple counters.

The trade-off: atomics work for primitive types and small operations.
Anything more complex (e.g., "increment a counter in a hash map") needs
a `Mutex`.

### When `RwLock` and not `Mutex`

A rule of thumb:

- Reads > Writes × 10: use `RwLock`.
- Writes frequent, or critical section large: use `Mutex`.
- Counter / flag: use atomics.
- Complex data with many fields: split into smaller locks, or use a
  single `Mutex` if contention is low.

Commonware's P2P uses `RwLock` for the active peer set (read on every
message), `Mutex` for per-peer state (write on every message), and
atomics for stats. The combination keeps contention low.

## OSI and TCP/IP, fully

The main-body section "The OSI model — where P2P actually lives"
introduces the 7-layer OSI model and the 4-layer TCP/IP model.
This section goes deeper: what each layer actually does, how
encapsulation works, where Commonware's P2P layer fits, and the
relationship between "transport" and "application."

### The 7-layer OSI model

The OSI (Open Systems Interconnection) model was developed by
ISO in the late 1970s as a reference model for network
protocols. It is *not* an implementation guide; it is a
conceptual framework. Real protocols (TCP, IP, HTTP) don't map
cleanly to the 7 layers, but the model is useful for thinking
about network communication.

The layers, from bottom to top:

1. **Physical.** The actual transmission of bits over a medium
   (copper wire, fiber, radio). Examples: Ethernet 10BASE-T
   (100 Mbps over twisted pair), 802.11ax (Wi-Fi 6), fiber
   optic links.

2. **Data link.** Framing of bits into "frames" with error
   detection. Examples: Ethernet (IEEE 802.3), Wi-Fi (IEEE
   802.11), PPP (Point-to-Point Protocol).

3. **Network.** Routing of packets between networks. Examples:
   IP (Internet Protocol), ICMP (for ping/traceroute).

4. **Transport.** End-to-end communication between processes.
   Examples: TCP (reliable byte stream), UDP (unreliable
   datagrams), QUIC (reliable byte stream over UDP).

5. **Session.** Management of connections between processes.
   Examples: TLS (sometimes), RPC frameworks (gRPC).

6. **Presentation.** Data representation, encryption, compression.
   Examples: TLS (often considered here), JSON encoders, ASN.1.

7. **Application.** User-facing protocols. Examples: HTTP, DNS,
   SMTP, SSH.

Note: TLS spans layers 5 and 6 in different models. Some books
put TLS at layer 4 (transport), others at layer 7 (application).
This is because TLS is *between* TCP and the application; it
provides encryption (presentation) and connection management
(session). Commonware's P2P layer is similar: it sits between
TCP and the application, providing authentication, encryption,
and rate limiting.

### The 4-layer TCP/IP model

The TCP/IP model is the *de facto* layering used in real
networks. It has four layers:

1. **Link.** Combines OSI's physical and data link layers.
   Examples: Ethernet, Wi-Fi.

2. **Internet.** Network layer. Example: IP.

3. **Transport.** Transport layer. Examples: TCP, UDP.

4. **Application.** Combines OSI's session, presentation, and
   application layers. Examples: HTTP, TLS, SSH, gRPC.

The TCP/IP model is less precise than the OSI model but more
reflective of how real protocols are structured. Most
practitioners use the TCP/IP layering and refer to OSI only when
they want to be precise.

### Encapsulation

When data moves down the stack, each layer adds its own header
(or trailer). This is **encapsulation**:

```
Application:  [HTTP request]
Transport:    [TCP header][HTTP request]
Internet:     [IP header][TCP header][HTTP request]
Link:         [Ethernet header][IP header][TCP header][HTTP request][Ethernet trailer]
```

Each layer treats the layer above's data as opaque payload. The
TCP header doesn't know what's inside the HTTP request; it just
treats it as bytes.

On the receiving end, each layer strips its own header and
passes the remaining bytes up the stack.

### Where P2P fits

Commonware's P2P layer sits *between TCP and the application*:

```
Application:    [consensus message]
P2P:            [P2P header][consensus message]
TLS/encryption: [TLS header][P2P header][consensus message]
Transport (TCP):[TCP header][TLS header][P2P header][consensus message]
Internet (IP):  [IP header][TCP header][TLS header][P2P header][consensus message][TCP trailer]
Link:           [Ethernet header][IP header]...[TCP trailer][Ethernet trailer]
```

The P2P header carries:

- Sender's public key (or a reference to it).
- Channel identifier (which "topic" the message belongs to).
- Payload length.
- Authenticated tag (the MAC over the message).

This is conceptually similar to TLS, but application-specific:
P2P knows about channels, public keys, and the Commonware
protocol.

### The role of ports

A "port" is a 16-bit number (1-65535) that identifies a
specific endpoint on a host. When a packet arrives at a host,
the IP address identifies the host, and the port number
identifies the application.

For example, your web server might listen on port 8080. When a
client sends an HTTP request to your server's IP address on port
8080, the OS routes the packet to the web server process.

Commonware's P2P layer typically listens on a configurable port
(default 8080 or whatever the application chooses). Each peer
connects to a specific IP:port combination.

### The difference between "transport" and "application"

The transport layer (TCP, UDP, QUIC) provides *byte transport*
between two endpoints. The application layer provides *meaning*
to those bytes.

For Commonware, the transport layer provides:

- Reliable (TCP) or unreliable (UDP) byte streams.
- Connection-oriented (TCP) or connectionless (UDP) semantics.

The application layer (Commonware's P2P) provides:

- Peer authentication (who is this?).
- Encryption (only the intended recipient can read).
- Rate limiting (don't overwhelm me).
- Channel multiplexing (multiple "topics" on one connection).
- Message framing (where does one message end and the next begin).

The transport doesn't know about any of this. It just moves
bytes.

### Why this matters

Understanding the layering is essential for debugging:

- If a connection is slow, is the problem at the link layer
  (poor Wi-Fi signal) or the transport layer (TCP backpressure)
  or the application layer (Commonware's rate limiter)?
- If messages are lost, is it the network dropping packets
  (link layer), TCP retransmits (transport), or Commonware
  rejecting messages (application)?

Each layer has its own metrics: link layer has signal strength
and retransmits, transport has RTT and cwnd, application has
queue depth and rate-limit drops.

Commonware's metrics are at the application layer. To diagnose
network issues, you need OS-level metrics (via `ss`, `netstat`,
`/proc/net/...`, or tools like `iftop` and `wireshark`).

### The TCP/IP stack in Commonware

Commonware uses the Tokio runtime (chapter 02), which uses the
operating system's TCP/IP stack. There is no user-space TCP in
Commonware. The stack is:

- Link: OS driver (Ethernet, Wi-Fi, etc.).
- Internet: OS IP stack.
- Transport: OS TCP (via `tokio::net::TcpStream`).
- Application: Commonware's P2P layer.

Some advanced networking libraries (like `quinn` for QUIC)
implement transport-layer protocols in user space, but they
still rely on the OS for IP and link layers. User-space TCP
exists (e.g., `smoltcp`) but is rare outside embedded systems.


## TCP in depth

The main-body section "TCP — reliable byte stream" introduces
TCP. This section goes deeper: the three-way handshake, sliding
window flow control, congestion control (slow start, congestion
avoidance, fast retransmit, fast recovery, CUBIC, BBR), and the
relationship between TCP and Commonware's "reliable or nullify"
pattern.

### The three-way handshake

Before any data is sent, TCP establishes a connection with a
three-way handshake:

```
Client                  Server
  |--- SYN seq=x -------->|
  |<-- SYN seq=y ACK=x+1 -|
  |--- ACK=y+1 ---------->|
  |                       |
  |--- data ------------->|
```

1. **SYN:** Client sends a SYN packet with an initial sequence
   number (ISN) `x`.
2. **SYN-ACK:** Server responds with a SYN packet (ISN `y`) and
   an ACK for `x + 1`.
3. **ACK:** Client ACKs the server's ISN (`y + 1`).

After this, both sides are in `ESTABLISHED` state and can send
data.

The three-way handshake is necessary because both sides need to
establish initial sequence numbers, and each side needs to
confirm that the other side is reachable. A two-way handshake
would not allow the server to confirm that the client received
its ISN.

The initial sequence numbers are randomized for security: an
attacker who can predict ISNs can inject packets into existing
connections (the "TCP sequence prediction attack").

### Sliding window flow control

TCP uses a **sliding window** to allow the sender to transmit
multiple packets before receiving an ACK. The receiver
advertises a window size (the amount of data it can buffer), and
the sender ensures that the number of unacknowledged bytes in
flight is at most the window size.

```
Sender:
   sent: |----|------|-----|-----|-----|-----|--|
                            ^                  ^
                            |                  |
                          send_base       send_base + window

Receiver:
   received: |--|--|--|--|--|--|--|--|--|
              ^            ^              ^
              |            |              |
            rcv_base   last_byte_read  last_byte_recv
```

As ACKs come in, `send_base` advances and the window slides
forward. The receiver can adjust the window size at any time
(flow control), and the sender respects it.

This is the foundation of TCP's reliability: every byte sent is
eventually ACKed, and the sender knows which bytes have been
received.

### Congestion control

The sliding window provides flow control (the receiver's
buffering limit). **Congestion control** is separate: it limits
the sender's rate based on *network conditions*.

The basic idea: if packets are being dropped, the network is
congested. Reduce the sending rate. If packets are arriving
without drops, the network has spare capacity. Increase the rate
slowly.

The key state variable is the **congestion window (cwnd)**, the
maximum number of bytes the sender can have in flight. The
sender's effective window is `min(receiver_window, cwnd)`.

### Slow start

When a connection begins, cwnd is set to a small value
(historically 1 segment, now ~10 segments in modern Linux). For
each ACK received, cwnd is increased by one segment. This is
**slow start**: cwnd grows exponentially (doubling per RTT).

Slow start is "slow" only in the sense that it's slower than
sending as fast as possible. In practice, cwnd reaches a usable
size within a few RTTs.

### Congestion avoidance

When cwnd reaches a **slow start threshold (ssthresh)**, the
algorithm transitions from slow start to **congestion avoidance**.
In congestion avoidance, cwnd grows linearly (one segment per
RTT, not one per ACK). This is much slower than slow start.

The intuition: in slow start, we're probing for the network's
capacity, so we grow fast. In congestion avoidance, we're at or
near capacity, so we grow slowly to avoid overshooting.

### Fast retransmit and fast recovery

When the sender detects a packet loss (typically via duplicate
ACKs — three duplicate ACKs indicate that one packet was lost
but later packets arrived), it doesn't wait for the timeout.
Instead:

1. **Fast retransmit:** the sender retransmits the lost packet
   immediately, without waiting for the timeout.
2. **Fast recovery:** the sender halves cwnd (instead of
   dropping it to 1 segment) and continues in congestion
   avoidance.

This is much faster than waiting for the timeout (which can be
hundreds of milliseconds).

### CUBIC

Modern Linux uses **CUBIC** as the default congestion control
algorithm (replacing the older Reno and Tahoe). CUBIC's key
idea: the cwnd growth function is a cubic function of time
since the last loss event, not a linear function of RTT.

The advantage: CUBIC is fair across connections with different
RTTs (Reno's linear growth favors low-RTT connections), and it
recovers faster from losses.

### BBR

**BBR (Bottleneck Bandwidth and Round-trip propagation time)**
is a more recent congestion control algorithm that doesn't
rely on packet loss as the primary signal. Instead, BBR
estimates the bottleneck bandwidth and the round-trip
propagation time, and uses these to set cwnd.

BBR is faster than CUBIC in many scenarios (especially
high-bandwidth, high-latency links), but it has been criticized
for being unfair to other congestion control algorithms.

### The relationship to Commonware's "reliable or nullify"

Commonware's P2P uses TCP underneath. TCP already provides
reliability: lost packets are retransmitted, and the receiver
sees a byte stream with no gaps.

But Commonware adds an application-level timeout: if a request
to a peer doesn't get a response within some time, the
application treats it as if the request was lost and "nullifies"
the request (i.e., gives up on it and proceeds with a default).

This is a useful abstraction: from the application's
perspective, every request either succeeds within a deadline or
is nullified. The application doesn't have to worry about
partial failures, retries, or message reordering.

The cost: if the TCP connection is slow (high RTT, congestion),
the application-level timeout may fire before the response
arrives. The application then nullifies a request that would
have succeeded if given more time.

This is the trade-off between "reliable in principle" (TCP
guarantees delivery) and "reliable in time" (application
deadlines). Commonware chooses the latter: if a request doesn't
get a response within the deadline, it's nullified, regardless
of whether the underlying TCP would have eventually delivered it.

### TCP's Nagle algorithm

The **Nagle algorithm** is a TCP optimization that reduces the
number of small packets. The idea: if you have a small packet to
send and there is unacknowledged data in flight, wait for the
ACK before sending. This coalesces small writes into larger
packets.

The downside: for latency-sensitive applications (interactive
gaming, RPC), the Nagle delay can add tens of milliseconds.
Commonware disables Nagle on its TCP sockets (`set_nodelay(true)`)
to ensure low-latency message delivery.

### TCP_CORK and TCP_QUICKACK

Linux has additional socket options for tuning:

- **TCP_CORK:** Coalesces multiple writes into a single packet
  even when there is no unacknowledged data in flight. Useful
  for batching small writes.
- **TCP_QUICKACK:** Disables delayed ACKs. Useful for
  latency-sensitive applications.

Commonware uses `TCP_NODELAY` (disabling Nagle) for all
connections. It does not typically set `TCP_CORK` or
`TCP_QUICKACK` because they add complexity for marginal gains.

### The TCP receive window and buffer sizes

The TCP receive window is the maximum amount of unacknowledged
data the sender can have in flight. By default, Linux uses an
auto-tuning algorithm that adjusts the window based on the
available buffer space and the path's bandwidth-delay product.

For high-bandwidth, long-latency links, the default buffer size
might be too small, limiting throughput. The `tcp_rmem` and
`tcp_wmem` sysctl variables control the buffer sizes.

Commonware exposes the buffer size via the `P2P` configuration,
allowing applications to tune for their specific deployment.

### The bandwidth-delay product

The **bandwidth-delay product (BDP)** is the maximum amount of
data that can be "in flight" at any time:

```
BDP = bandwidth × RTT
```

For a 1 Gbps link with 100 ms RTT:

```
BDP = 1 Gbps × 100 ms = 1e9 bits/s × 0.1 s = 1e8 bits = 12.5 MB
```

To fully utilize the link, the TCP receive window must be at
least 12.5 MB. If the window is smaller, the sender will block
waiting for ACKs, and throughput will be limited.

For inter-continental links (e.g., US to Europe, ~80 ms RTT) and
high-bandwidth connections (10 Gbps), the BDP can be hundreds of
MB. This is why **TCP window scaling** (RFC 7323) was
introduced: to allow windows larger than 64 KB.

### TCP and TLS

Commonware's P2P uses TCP for transport, but the P2P layer
itself provides the encryption (via X25519 + ChaCha20-Poly1305,
covered in the next section). TLS is *not* used because:

1. Commonware's P2P needs to know the peer's public key for
   authentication, not just the host. TLS authenticates the host
   via certificates, which is a different model.
2. The P2P layer is application-specific (channels, rate limits)
   and needs to be in user space.

But Commonware could layer TLS on top of TCP if the deployment
required it (e.g., to traverse corporate firewalls that expect
TLS). The TLS handshake would be performed first, then the P2P
handshake inside the TLS tunnel.


## UDP in depth

The main-body section "UDP — best-effort datagrams" introduces
UDP. This section goes deeper: the UDP header, the semantics of
unreliable delivery, and the cases where UDP is the right
choice.

### The UDP header

UDP is a thin layer over IP. The header is just 8 bytes:

```
+------+------+--------+
| src  | dst  | length |
| port | port |        |
+------+------+--------+
| check- |       |
| sum    |       |
+--------+-------+
```

- Source and destination ports (16 bits each).
- Length (16 bits, including header).
- Checksum (16 bits, optional in IPv4, mandatory in IPv6).

UDP doesn't establish a connection, doesn't retransmit lost
packets, and doesn't guarantee ordering. The application is
responsible for all of that if it needs it.

### Why UDP is unreliable by design

UDP is unreliable because adding reliability requires state
(sequence numbers, ACK tracking, retransmission timers), and
state is the enemy of simplicity.

TCP's reliability machinery is ~10,000 lines of code in the
Linux kernel. UDP's machinery is ~1,000 lines. If your
application doesn't need the full TCP feature set, UDP is much
simpler.

For example, a DNS query is one packet out, one packet back. If
the response is lost, the application retries. No need for TCP's
overhead.

### When UDP is the right choice

UDP is right when:

1. **Low latency is critical.** TCP's retransmits add latency.
   For real-time applications (voice, video, gaming), UDP's
   "best effort" is better than TCP's "guaranteed but late."
2. **The application handles reliability itself.** Some
   applications have their own retransmit logic that is
   tailored to the use case (e.g., real-time ARQ for video).
3. **Multicast or broadcast.** TCP is point-to-point. UDP
   supports multicast and broadcast, which TCP does not.
4. **Stateless servers.** DNS, NTP, and other stateless services
   use UDP because there's no per-connection state to maintain.
5. **The message fits in one packet.** If every message is small
   and you don't need flow control, UDP is simpler than TCP.

Commonware uses TCP because the messages are part of a BFT
consensus protocol that needs reliability and ordering. UDP is
the wrong choice here.

### The QUIC story

QUIC (RFC 9000) is a transport protocol that runs over UDP and
provides TCP-like reliability with additional features:

- **Stream multiplexing** without head-of-line blocking.
- **Built-in TLS 1.3.**
- **0-RTT handshake** (resumed connections).
- **Connection migration** (changing IP/port without dropping
  the connection).
- **User-space implementation** (QUIC is in user space, not the
  kernel).

QUIC is the basis for HTTP/3 (RFC 9114). It is increasingly
common for latency-sensitive applications.

Commonware does not currently use QUIC, but the design of the
P2P layer is amenable to QUIC: the P2P layer sits above the
transport, and replacing TCP with QUIC is mostly a matter of
swapping `tokio::net::TcpStream` for a QUIC stream type.

### UDP and DDoS

UDP is the protocol of choice for **amplification DDoS attacks**.
The attacker sends a small request to a server with a spoofed
source IP, and the server sends a large response to the victim's
IP. Common amplification vectors:

- **DNS:** A 60-byte query can produce a 4,000-byte response.
- **NTP:** A small request can produce a large response.
- **Memcached:** Historically, a 15-byte request could produce
  a 750 KB response.

For Commonware, this is a concern at the network edge: an
attacker could spoof UDP packets to your Commonware node, but
since Commonware uses TCP, this is not directly applicable. The
risk is if you layer QUIC on top of UDP and the QUIC handshake
is exploited for amplification.

The mitigation: rate limiting at the network edge, and
authentication of incoming packets (Commonware's P2P does this
via the X25519 handshake).

### UDP hole punching

**UDP hole punching** is a technique for establishing a peer-to-
peer UDP connection through a NAT. The idea:

1. Both peers send UDP packets to a rendezvous server.
2. The server tells each peer the other's public IP:port.
3. Both peers send UDP packets directly to each other.
4. The NATs create "holes" — temporary mappings from the
   public port to the private endpoint.
5. Subsequent packets from the peer are forwarded through the
   hole.

This is the basis for WebRTC and many P2P systems.

Commonware doesn't use UDP hole punching (it uses TCP via a
bootstrapper). But the principle is interesting for any P2P
system that wants to avoid a central relay.

### UDP packet size

The maximum UDP packet size is 65,507 bytes (the IP packet
size minus the IP and UDP headers). On most networks, the
**MTU** (Maximum Transmission Unit) is 1,500 bytes, so UDP
packets larger than ~1,472 bytes will be fragmented. Fragmented
UDP is unreliable: if any fragment is lost, the entire packet
is lost.

Commonware's P2P uses TCP, so packet size is not a concern at
the UDP level. If you used UDP directly, you would want to keep
messages under 1,400 bytes to avoid fragmentation.


## Authenticated encryption in depth

The main-body section "Authenticated encryption — the AEAD
concept" introduces AEADs. This section goes deeper: the
construction of ChaCha20-Poly1305 and AES-GCM, the role of
associated data, the security model (IND-CCA2), and the
practical considerations.

### What AEAD gives you

An **AEAD (Authenticated Encryption with Associated Data)**
provides:

1. **Confidentiality:** The plaintext is hidden from anyone who
   doesn't have the key.
2. **Authenticity:** The ciphertext was produced by someone
   with the key (and the associated data).
3. **Integrity:** The ciphertext has not been modified.

This is more than just encryption (which only provides
confidentiality). Without authentication, an attacker can
modify the ciphertext in ways that produce predictable changes
in the plaintext, and you won't know.

The "associated data" is the fourth input to the AEAD: it's
authenticated (the receiver can verify it wasn't modified) but
not encrypted (it's sent in the clear). For Commonware's P2P,
the associated data is the peer's identity (their public key or
a derivative).

### The security model: IND-CCA2

The standard security notion for AEADs is **IND-CCA2**
(Indistinguishability under Adaptive Chosen Ciphertext Attack).
Informally:

- The attacker can submit chosen plaintexts and receive
  ciphertexts.
- The attacker can submit chosen ciphertexts and receive
  decryptions (except for a "challenge" ciphertext).
- The attacker then submits two plaintexts (m₀, m₁), receives a
  ciphertext for one of them (chosen at random), and must guess
  which one.

IND-CCA2 security means the attacker cannot do better than
random guessing, even with access to the decryption oracle.

Both ChaCha20-Poly1305 and AES-GCM are IND-CCA2 secure under
standard assumptions (the security of the underlying cipher and
hash).

### ChaCha20 internals

ChaCha20 is a stream cipher designed by Daniel Bernstein in
2008. It operates on a 4×4 matrix of 32-bit words (the "state")
and produces a stream of pseudo-random bytes by repeatedly
applying a quarter-round function:

```
quarter_round(a, b, c, d):
    a += b;  d ^= a;  d = rotl(d, 16)
    c += d;  b ^= c;  b = rotl(b, 12)
    a += b;  d ^= a;  d = rotl(d, 8)
    c += d;  b ^= c;  b = rotl(b, 7)
```

A full ChaCha20 block consists of 20 rounds (10 "double rounds")
of these operations, applied to all four columns and all four
diagonals of the state matrix.

The state is initialized from:

- A 256-bit key.
- A 96-bit nonce.
- A 32-bit counter (incremented for each block).

ChaCha20 is fast on platforms without AES hardware acceleration
(older ARM, mobile), and has constant-time implementations.

### Poly1305 internals

Poly1305 is a **universal hash-based MAC** designed by Bernstein
in 2005. The construction:

1. Interpret the message as a sequence of 16-byte blocks.
2. Compute the polynomial:
   ```
   p(x) = Σ m_i * x^(n-i)  (mod 2^130 - 5)
   ```
   where m_i are the message blocks and x is a key-dependent
   value.
3. Compute:
   ```
   mac = (p(x) + s) mod 2^128
   ```
   where s is a key-dependent value chosen to make this a
   one-time MAC.

The output is a 16-byte MAC.

Poly1305 is a *one-time* MAC: it is secure only for a single
message. Using the same key and nonce for two messages leaks
information. To make it multi-message secure, ChaCha20-Poly1305
uses a unique nonce for each message.

### ChaCha20-Poly1305 as an AEAD

The construction:

1. Generate a Poly1305 key by running ChaCha20 once with the
   AEAD nonce and a counter of 0.
2. Encrypt the plaintext with ChaCha20 starting at counter 1.
3. Compute the Poly1305 MAC over the AAD || ciphertext, using
   the Poly1305 key from step 1.
4. Output ciphertext || MAC.

The nonce is a 96-bit value that must be unique for each
encryption with the same key. Commonware uses a counter for the
nonce: the first message has nonce 0, the second has nonce 1,
etc.

### AES-GCM internals

AES-GCM is another widely-used AEAD. The construction:

1. Encrypt the plaintext with AES in CTR mode (AES-CTR).
2. Compute a **GHASH** over the AAD || ciphertext.
3. Encrypt the GHASH result with AES-CTR using a key derived
   from the AES-GCM key.

GHASH is a universal hash in the field GF(2^128), with the
multiplication-by-H construction.

AES-GCM is fast on platforms with AES hardware acceleration
(modern x86 with AES-NI, modern ARM). Without hardware
acceleration, AES-GCM is slower than ChaCha20-Poly1305.

### The choice of AEAD

Commonware uses ChaCha20-Poly1305 because:

1. **Constant-time on all platforms.** ChaCha20 is straightforward
   to implement in constant time. AES-GCM implementations can
   leak via cache timing if not carefully written.
2. **Fast without hardware acceleration.** On older mobile
   devices or ARM servers without AES-NI, ChaCha20-Poly1305 is
   faster than AES-GCM.
3. **Simpler implementation.** ChaCha20-Poly1305 has fewer lines
   of code than AES-GCM and is easier to audit.

If you're deploying on modern x86 with AES-NI, AES-GCM is
slightly faster. But the difference is small (10-20%).

### The role of associated data

The associated data is *authenticated but not encrypted*. In
Commonware's P2P, the associated data is the peer's public key
(or a derivative). This binds the ciphertext to the peer's
identity: an attacker who tries to replay a ciphertext from
peer A to peer B will fail the authentication check, because
the associated data (peer B's public key) won't match.

This is the same trick TLS uses with the "channel binding" or
"ALPN" extension.

### Nonce reuse

Both ChaCha20-Poly1305 and AES-GCM *fail catastrophically* if a
nonce is reused with the same key:

- For ChaCha20-Poly1305, nonce reuse leaks the XOR of the two
  plaintexts (since ChaCha20 is a stream cipher).
- For AES-GCM, nonce reuse leaks the authentication key.

Commonware uses a counter for the nonce. The counter starts at 0
and increments for each message. If the counter overflows (after
2^96 messages, which is "never"), the connection is closed and a
new handshake is performed.

### Replay attacks

An AEAD protects against message modification but *not* against
replay (re-sending a captured ciphertext). The receiver needs
additional logic to reject replays:

- A **sequence number** in the message.
- A **sliding window** of recently-seen sequence numbers.
- A **time-based** rejection (reject messages older than some
  threshold).

Commonware's P2P uses a sequence number per message, and the
receiver rejects messages with sequence numbers outside a
sliding window.

### Putting it together

AEADs are the workhorse of modern authenticated communication.
Commonware uses ChaCha20-Poly1305 because it is:

- Fast on all platforms.
- Constant-time.
- Simpler to audit than AES-GCM.

The trade-offs (slightly slower on platforms with AES-NI) are
acceptable for the security guarantees.


## X25519 + ChaCha20-Poly1305 handshake in depth

The main-body section "The X25519 + ChaCha20-Poly1305 handshake"
introduces the basic flow. This section goes deeper: the
Curve25519 elliptic curve, the X25519 key agreement protocol,
the HKDF step, ephemeral keys, and perfect forward secrecy.

### The Curve25519 elliptic curve

Curve25519 is a Montgomery-form elliptic curve:

```
y² = x³ + 486662 x² + x
```

over the prime field F_p where:

```
p = 2^255 - 19
```

The curve was designed by Daniel Bernstein in 2006. Properties:

- **255-bit security.** The DLP on Curve25519 is as hard as
  factoring a ~3072-bit RSA modulus.
- **Fast scalar multiplication.** The Montgomery ladder is
  constant-time and runs in ~250,000 cycles on a modern CPU.
- **No cofactor issues.** The curve has cofactor 8, but the
  X25519 protocol works in the prime-order subgroup, so this
  doesn't matter for security.
- **No invalid-curve attacks.** The curve has twist security,
  so even if a point is on the twist, it doesn't leak the
  scalar.

Curve25519 is used for Diffie-Hellman key exchange, not for
signatures. The signature scheme on the same curve is Ed25519
(Edwards form).

### X25519 key agreement

The **X25519** protocol computes a shared secret from two
peers' keys:

1. Alice has a 32-byte private key `a` (a random scalar).
2. Alice computes her public key `A = a * G` (a 32-byte point).
3. Alice sends `A` to Bob.
4. Bob has a 32-byte private key `b` and public key `B = b * G`.
5. Bob sends `B` to Alice.
6. Alice computes `a * B` (using her private key and Bob's
   public key).
7. Bob computes `b * A` (using his private key and Alice's
   public key).
8. Due to the group law, `a * B = a * b * G = b * a * G = b * A`.
9. Both have the same shared secret.

The shared secret is a 32-byte point on the curve. This is then
fed into HKDF to derive encryption keys.

### The HKDF step

**HKDF (HMAC-based Key Derivation Function)** is a two-step
process:

1. **Extract:** Given the input keying material (IKM) and an
   optional salt, produce a pseudorandom key (PRK) using HMAC:
   ```
   PRK = HMAC-SHA256(salt, IKM)
   ```
2. **Expand:** Given the PRK and an info string, produce output
   keying material (OKM) of arbitrary length:
   ```
   OKM = HMAC(PRK, "" || info || 0x01) ||
         HMAC(PRK, OKM_1 || info || 0x02) ||
         ...
   ```

For Commonware's P2P, the IKM is the X25519 shared secret, and
the info string includes the handshake transcript (so the keys
are bound to the handshake). The OKM is split into:

- A 32-byte encryption key (for ChaCha20).
- A 32-byte MAC key (for Poly1305).
- A 32-byte IV seed (for nonce generation).

### Ephemeral keys and PFS

The handshake above uses **static keys** (Alice and Bob's long-
term identities). This provides authentication (Alice knows
she's talking to Bob, and vice versa), but **not** perfect
forward secrecy (PFS).

**PFS** means: even if a long-term private key is compromised
in the future, past sessions remain secure. Without PFS, if an
attacker records encrypted traffic today and later compromises
Alice's long-term key, they can decrypt the recorded traffic.

To achieve PFS, each handshake uses **ephemeral keys** that are
generated fresh and discarded after the handshake:

1. Alice generates an ephemeral key pair (e_a, E_a) where E_a =
   e_a * G.
2. Bob generates an ephemeral key pair (e_b, E_b).
3. Both sides compute two shared secrets: one from the static
   keys, one from the ephemeral keys.
4. The session keys are derived from a combination of both
   shared secrets (typically XOR or HKDF over both).

After the handshake, Alice and Bob delete their ephemeral
private keys (e_a, e_b). Now, even if an attacker later
compromises Alice's long-term key, they cannot recompute the
ephemeral shared secret, so they cannot decrypt the recorded
session.

Commonware's P2P uses **ephemeral keys for every connection**.
This is what gives Commonware's connections perfect forward
secrecy.

### The handshake transcript

The "transcript" of the handshake is a hash of all the messages
exchanged during the handshake. Commonware includes the
transcript in the HKDF info string, so the derived keys are
bound to the exact handshake that produced them.

This prevents **transcript collision attacks**: an attacker who
can produce two different handshakes that hash to the same
transcript cannot trick a peer into reusing the same session
keys.

### The role of a "binder"

Some handshakes include a "binder": a MAC over the transcript
that is verified before the handshake completes. This catches
attackers who try to inject handshake messages.

Commonware's handshake includes a binder. The receiver computes
the expected binder from their long-term private key and the
received ephemeral public key; if the binder doesn't match, the
handshake is aborted.

### Why X25519, not ECDH over a NIST curve

Commonware uses X25519 instead of ECDH over P-256 (the NIST
standard curve) because:

1. **Better performance.** X25519 is faster than P-256 on most
   platforms.
2. **No parameter validation issues.** P-256 has a history of
   "invalid curve" attacks where implementations accepted points
   not on the curve, leaking the scalar. X25519 has twist
   security, so this isn't an issue.
3. **Cleaner implementation.** X25519 is easier to implement
   correctly than P-256.
4. **Constant-time by design.** X25519's Montgomery ladder is
   naturally constant-time. P-256 implementations must be
   carefully written to be constant-time.

The trade-off: P-256 is the NIST standard, and some compliance
regimes require it. If you're deploying in such a regime, you'd
use P-256; otherwise, X25519 is preferred.

### Putting it together

The X25519 + ChaCha20-Poly1305 handshake provides:

- **Mutual authentication** via static keys.
- **Forward secrecy** via ephemeral keys.
- **Confidentiality and integrity** via the ChaCha20-Poly1305
  AEAD.
- **Replay resistance** via sequence numbers in the AEAD layer.

This is the same set of properties as TLS 1.3 with X25519 and
ChaCha20-Poly1305, but tailored for Commonware's specific
needs (peer-to-peer rather than client-server).

The handshake completes in **1 round trip**: Alice sends her
ephemeral key, Bob sends his, both derive the same keys, and
data flows. This is the same as TLS 1.3's 1-RTT handshake.

### What if a peer is offline during the handshake?

If a peer is offline (or unreachable) when you try to handshake,
the handshake times out. The application treats this as a
connection failure and may retry or nullify the operation.

Commonware's handshake has a default timeout (typically 5-10
seconds). If the handshake doesn't complete in that time, the
connection is aborted.

### What if the handshake produces bad keys?

The HKDF step includes a sanity check: the derived keys are
checked for "low order" or other known-bad values. If the
check fails, the handshake is aborted and the connection is
closed.

This is a defense against implementation bugs: if the X25519
implementation produced a degenerate output (e.g., the all-zero
shared secret, which is technically possible with negligible
probability), the connection would be vulnerable. The sanity
check catches this.


## NAT traversal in depth

The main-body section "NAT traversal — STUN, TURN, ICE, and the
bootstrapper pattern" introduces the four NAT-traversal
technologies. This section goes deeper: why NATs exist, how each
technology works, and why Commonware chose the bootstrapper
pattern.

### Why NATs exist

A **NAT (Network Address Translation)** is a function performed
by a router that maps private IP addresses to public IP
addresses. The most common form is **PAT (Port Address
Translation)**, where many private IPs share one public IP, with
different port numbers distinguishing them.

NATs exist because IPv4 addresses are scarce. There are ~4.3
billion IPv4 addresses, but the world has many more devices. NATs
let a household or business share one public IP among many
devices.

The downside: NATs break the end-to-end principle of the
internet. From the outside, you cannot initiate a connection to
a device behind a NAT — the NAT doesn't know which internal
device should receive the packet. This is a problem for
peer-to-peer protocols, where peers need to accept incoming
connections from anyone.

### The four types of NAT

There are four main types of NAT, in increasing order of
difficulty for traversal:

1. **Full cone NAT.** Any external host can send packets to the
   internal host via the mapped public IP:port. (Rare in
   practice.)

2. **Restricted cone NAT.** Only external hosts that the
   internal host has sent packets to can send packets back.

3. **Port-restricted cone NAT.** Only external hosts that the
   internal host has sent packets to *from a specific port* can
   send packets back.

4. **Symmetric NAT.** Each outbound connection gets a different
   public port. (Most restrictive; common in corporate networks.)

The more restrictive the NAT, the harder it is to traverse.

### STUN (Session Traversal Utilities for NAT)

**STUN** is a protocol for discovering the public IP:port that
a NAT has assigned to a client:

1. The client sends a STUN binding request to a STUN server.
2. The server responds with the public IP:port it saw the
   request coming from.
3. The client now knows its external address.

This works for full cone NATs and restricted cone NATs, but not
for symmetric NATs (where the mapped port depends on the
destination).

STUN is a small, simple protocol defined in RFC 5389. It runs
over UDP, and the binding request is just a few bytes.

### TURN (Traversal Using Relays around NAT)

**TURN** is a protocol for relaying traffic through a server when
direct traversal is impossible:

1. The client allocates a relay address on the TURN server.
2. The client sends the relay address to its peer (via a
   signaling channel).
3. The peer sends traffic to the relay address, and the TURN
   server forwards it to the client.

TURN works for all NAT types, including symmetric NATs. The
downside: all traffic goes through the relay, which adds latency
and bandwidth cost.

TURN is defined in RFC 5766. It is used by WebRTC as a fallback
when direct connections fail.

### ICE (Interactive Connectivity Establishment)

**ICE** is a protocol that combines STUN and TURN:

1. Each peer gathers a list of candidate addresses (local,
   server-reflexive from STUN, relayed from TURN).
2. Each peer sends its candidates to the other via a signaling
   channel.
3. Each peer tries all pairs of candidates (local + remote),
   performing connectivity checks.
4. The first pair that succeeds is used for the connection.

ICE works for all NAT types, but the signaling channel (for
exchanging candidates) requires a separate mechanism (e.g., a
SIP server, or manual configuration).

ICE is defined in RFC 8445. It is the standard NAT traversal
mechanism for WebRTC.

### The bootstrapper pattern

Commonware does not use STUN, TURN, or ICE. Instead, it uses a
**bootstrapper**: a known peer (or set of peers) that helps
newcomers join the network.

The flow:

1. New node knows the bootstrapper's IP:port (from
   configuration).
2. New node connects to the bootstrapper (TCP, since the
   bootstrapper has a public IP).
3. Bootstrapper tells the new node about other peers (via a
   gossip protocol or a known peer list).
4. New node connects to the other peers directly (via TCP).
5. New node is now part of the network.

This works for all NAT types as long as the new node can make
outgoing TCP connections. The bootstrapper is a public-facing
peer; the other peers may or may not be public-facing, but as
long as the new node can reach them, the connection works.

### Why the bootstrapper is right for BFT

For a BFT consensus protocol, NAT traversal is less of a
concern than for, say, a video conferencing app. The reasons:

1. **Stable membership.** BFT protocols have a known set of
   validators (the "validator set"). New validators are added
   via governance, not spontaneously.

2. **Public infrastructure.** Validators in production are
   typically deployed on cloud VMs with public IPs. NATs are not
   the norm.

3. **Operator-controlled configuration.** Operators can
   configure their bootstrapper and known peers explicitly.

4. **Predictable topology.** The network topology is known
   ahead of time, so NAT traversal isn't needed.

For consumer applications (where peers come and go, and NATs
are common), ICE is the right choice. For BFT validators
(deployed by experienced operators with public infrastructure),
the bootstrapper is simpler and sufficient.

### The cost of the bootstrapper

The bootstrapper is a single point of failure: if the
bootstrapper is down, new nodes cannot join. Mitigations:

1. **Multiple bootstrappers.** Commonware can be configured
   with a list of bootstrappers. If one is down, the new node
   tries the next.

2. **Bootstrapper as a service.** The bootstrapper can be
   operated as a public service (like a DNS server) by a
   trusted party.

3. **Self-bootstrap.** Existing peers can act as bootstrappers
   for new peers, eliminating the single point of failure.

For Commonware, the typical setup is:

- 3-5 well-known bootstrappers operated by the validator set.
- Each new node tries all bootstrappers in order until one
  responds.
- Once connected to one bootstrapper, the new node learns
  about all other peers.

This is robust enough for production deployments.

### Putting it together

NATs exist because of IPv4 address scarcity. STUN, TURN, and
ICE are the standard traversal mechanisms, but they require
sophisticated signaling and may not work for all NAT types.

Commonware's bootstrapper pattern is simpler: new nodes connect
to a known bootstrapper, learn about other peers, and connect
to them directly. This works for BFT validators (which are
typically deployed on public infrastructure) and avoids the
complexity of ICE.

For consumer applications or other peer-to-peer scenarios where
NATs are common, ICE is the right choice. Commonware's design
is not appropriate for those scenarios, but it is appropriate
for BFT consensus.


## Gossip protocols in depth

Commonware uses gossip for peer discovery and message
dissemination. This section covers the three main gossip
strategies (anti-entropy push, pull, push-pull), the trade-offs,
and when each is right.

### The problem

You have N peers, each with some local state. You want all peers
to converge to the same state (or to learn each other's state)
as quickly as possible, using minimal bandwidth.

This is the gossip problem. There are three main strategies:

1. **Push (Eager push / rumor mongering):** When you receive a
   new piece of state, push it to all peers. Each receiving peer
   pushes it to all of its peers, etc.

2. **Pull (Anti-entropy):** Periodically, ask peers for their
   state. They respond with their state, and you merge.

3. **Push-pull:** A combination: push new state, and pull to
   discover missed state.

Each strategy has different bandwidth and convergence
properties.

### Push (rumor mongering)

In push, when you learn a new piece of state (a "rumor"), you
push it to a random subset of peers. Those peers do the same.

Properties:

- **Fast initial spread.** In the first few rounds, the rumor
  reaches many peers quickly.
- **Slow tail.** The last few peers are hard to reach because
  the rumor's spread becomes probabilistic.
- **No guarantee of full convergence.** There's always a chance
  some peer never receives the rumor.
- **Bandwidth proportional to message count.** Each message
  triggers pushes to multiple peers.

For N peers, the push protocol's convergence time is O(log N)
rounds (each round, the number of informed peers doubles until
it reaches N).

### Pull (anti-entropy)

In pull, each peer periodically selects a random peer and asks
"what state do you have?" The peer responds with its state, and
you merge.

Properties:

- **Slow initial spread.** Each peer must be queried to
  discover new state.
- **Strong convergence.** Eventually, every peer knows every
  other peer's state (because each peer is queried
  periodically).
- **Bandwidth proportional to peer count, not message count.**
  Each pull request gets a response, but the responses are
  deduplicated.

For N peers, the pull protocol's convergence time is O(log N)
rounds (each round, the number of peers that have heard about
the latest state grows).

### Push-pull

Push-pull combines push and pull:

- Push new state to peers (fast initial spread).
- Pull from peers periodically to discover missed state (strong
  convergence).

This is the best of both worlds: fast initial spread and strong
convergence. The cost is more bandwidth: each round has both
push and pull traffic.

For Commonware's broadcast protocol, push-pull is the right
choice: fast spread is critical for low latency, and strong
convergence is critical for safety (every honest peer must
eventually see the message).

### Bandwidth vs time trade-offs

The three strategies have different points on the bandwidth/
time trade-off curve:

| Strategy | Bandwidth | Convergence time |
|----------|-----------|------------------|
| Push     | High      | Fast initial, slow tail |
| Pull     | Low       | Slow initial, strong final |
| Push-pull| Highest   | Fast initial, strong final |

For BFT consensus, push-pull is preferred: the cost is bandwidth,
but you can't afford to leave peers behind (a missed message is
a safety violation).

For metrics or telemetry (where missing a single sample is OK),
push or pull alone may suffice.

### The bandwidth formula

For N peers, each peer sends k push messages per round on
average, the total bandwidth per round is:

```
B = N * k * (message_size)
```

For N = 1000 peers, k = 4 pushes per round, and message_size =
1 KB, that's 4 MB per round. At 10 rounds per second, that's
40 MB/s of gossip bandwidth. Significant but manageable.

For Commonware, the broadcast protocol uses push-pull with
limits: each peer pushes to at most k peers per round, and the
"round" is event-driven (not periodic).

### Membership gossip

For peer discovery (learning who else is in the network),
Commonware uses a separate gossip channel for membership:

- Each peer periodically gossips its known peer set.
- When a peer learns about a new peer, it adds the new peer to
  its known set and gossips it.

This is a pull-style gossip (because membership is slow-changing
and convergence matters more than latency).

### Failure detection

Gossip can also be used for failure detection: each peer
gossips "I'm alive" periodically. If you don't hear from a peer
for some time, you suspect it's down.

This is the basis for protocols like SWIM (Scalable
Weakly-consistent Infection-style process group Membership).
Commonware's P2P uses a similar approach for tracking which
peers are reachable.

### The Commonware gossip implementation

Commonware's P2P layer uses:

- **Push-pull gossip** for application messages (broadcast).
- **Pull gossip** for membership (peer set updates).
- **Periodic "I'm alive"** for failure detection.

The implementation is in the `p2p::gossip` module. The exact
behavior is configurable: how many peers to push to per round,
how often to pull, how to handle missed messages, etc.

### Putting it together

Gossip is the workhorse of large-scale peer-to-peer protocols.
Commonware uses push-pull for application messages (where
convergence matters), pull for membership (where convergence
matters more than latency), and periodic "I'm alive" for
failure detection.

The choice of gossip strategy is a trade-off between bandwidth
and convergence properties. For BFT consensus, push-pull is
the right choice because the safety properties require strong
convergence.

## Appendix A — The handshake, in detail

The authenticated P2P layer uses `commonware-stream::encrypted` for the
transport. That library wraps TCP with:

1. **X25519 key exchange** — derive a shared secret.
2. **HKDF** — extract a symmetric key from the shared secret.
3. **ChaCha20-Poly1305** — encrypt all subsequent traffic.
4. **Nonce counter** — prevent nonce reuse across messages.

The handshake flow (from `commonware-stream/src/encrypted/handshake.rs`):

```
Dialer                                                 Listener
  |--- TCP SYN --->                                      |
  |<--- TCP SYN-ACK ---                                  |
  |--- TCP ACK --->                                      |
  |                                                      |
  |--- (noise pattern) ephemeral_pk, encrypted_payload --->|
  |<--- ephemeral_pk, encrypted_payload ---               |
  |                                                      |
  Both sides derive shared_secret from:
    shared_secret = X25519(ephemeral_sk, peer_ephemeral_pk)
  Then HKDF to get symmetric_key
                                                      
  Both sides verify peer signed with expected long-term pubkey
```

Properties:

- **Forward secrecy** — ephemeral keys are destroyed after handshake.
  Compromise of long-term keys doesn't decrypt past sessions.
- **Mutual authentication** — both sides prove knowledge of their long-term
  private key (via signature in the handshake payload).
- **Replay resistance** — the handshake payload includes a timestamp;
  peers reject greetings older than `max_handshake_age` (default 10s).

The dialer publishes its public key as part of its `Greeting`. The
listener knows which peer is dialing (via the signed greeting) and
verifies against its expected peer set.

### The `Info` struct

`p2p/src/authenticated/discovery/types.rs` (look for `Info`):

```rust
pub struct Info<C: PublicKey> {
    pub public_key: C,
    pub addresses: Vec<Ingress>,    // where this peer is reachable
    pub timestamp: SystemTime,
    pub signature: Signature,        // over (pk || addresses || timestamp)
}
```

This is the **greeting payload**. Contains everything a peer needs to
know:
- Who you are (public key).
- Where you can be reached (addresses).
- When the greeting was created (timestamp, for replay protection).
- Proof that you really are who you claim (signature).

The signature scheme is whatever `Signer` you configured (typically ed25519).

## Appendix B — Channel registration, in detail

When you register a channel, here's what happens under the hood
(`p2p/src/authenticated/discovery/channels.rs` and `lookup/channels.rs`):

```rust
network.register(channel_id: u64, quota: Quota, max_inflight: usize)
    -> (Sender, Receiver)
```

The `Sender` and `Receiver` are **handles** to the channel. Internally:

1. A new task is spawned to handle this channel's send/recv over all
   active connections to active peers.
2. The `Sender` is a `LimitedSender` (chapter 05 main text) — it checks
   per-peer quotas before sending.
3. The `Receiver` is a `Receiver` (chapter 05 main text) — receives
   `(peer_pk, IoBuf)` tuples from any active peer.

The `quota` is per-peer. If peer X is sending at the limit, your `Sender`
to peer X refuses (or backpresses, depending on policy). Messages to
peer Y still go through.

The `max_inflight` is the per-channel mailbox size (chapter 15). If the
receiver's mailbox is full, senders see `Feedback::Backoff`.

### Channel routing

When a `Data { channel: N, message: ... }` arrives from peer X:

1. The network layer looks up channel N in its registry.
2. If registered, hands the message to the channel's mailbox.
3. The channel's `Receiver::recv()` returns `(X, message)` when called.

If the channel isn't registered, the message is **dropped** and the
sender is logged. Commonware fails closed on unknown channels — if
you didn't register it, you don't want it.

## Appendix C — The connection state machine

Each authenticated connection has a state:

```
        ┌─────────┐
        │  Init   │
        └────┬────┘
             │ accept()
             ▼
        ┌─────────┐
        │  Wait   │  (waiting for peer's Greeting)
        │ Greet   │
        └────┬────┘
             │ recv(Greeting)
             ▼
        ┌─────────┐
        │ Send    │  (send our Greeting)
        │ Greet   │
        └────┬────┘
             │ send(Greeting)
             ▼
        ┌─────────┐
        │ Active  │  (encrypted channel ready)
        └────┬────┘
             │ peer disconnects / handshake timeout / protocol error
             ▼
        ┌─────────┐
        │  Dead   │  (cleanup, no more messages)
        └─────────┘
```

The Active state lasts until:
- Peer sends EOF or disconnect.
- Idle timeout (no traffic for X seconds).
- Protocol error (malformed message, signature failure).
- Local code closes the channel (e.g., during shutdown).

For each Active connection, the P2P layer maintains:
- The encrypted stream (from `commonware-stream`).
- Per-channel send/receive tasks.
- Statistics (bytes sent/recv, last activity time).

## Appendix D — The `discovery` actor system

`p2p/src/authenticated/discovery/actors/`. Multiple specialized actors:

| Actor | Job |
|---|---|
| `spawner` | Spawns connections to peers (dialer). |
| `ingress` | Handles incoming connections (listener). |
| `arbitrator` | Routes messages to the right channel. |
| `blocker` | Tracks blocked peers, refuses new connections. |
| `shutdown` | Coordinates graceful shutdown. |

The `spawner` actor owns the dialing logic:
- Look up which peers are in the active set.
- For each peer not currently connected, attempt to dial.
- Backoff on failure (exponential, capped at some max).
- Honor `peer_connection_cooldown` (don't hammer a peer that's down).

The `ingress` actor owns the listening logic:
- Accept incoming connections.
- Do the handshake.
- Hand off the encrypted stream to the arbitrator.

The `arbitrator` actor owns the routing:
- Multiplexes multiple channels over multiple connections.
- Tracks which peer sent which message.
- Applies rate limits per channel per peer.

## Appendix E — The `lookup` actor system (simpler)

`p2p/src/authenticated/lookup/`. Same actors, but simpler because
addresses are known ahead of time:

- No `bootstrappers` (you give the full peer list).
- No `Peers` gossip messages (everyone knows everyone).
- No `BitVec` ping (not needed; connections are stable).

Lookup is appropriate for **closed membership** networks — BFT
consensus where the validator set is known in advance.

## Appendix F — The `Recipients` semantics, in depth

`p2p/src/lib.rs:42-47`:

```rust
pub enum Recipients<P: PublicKey> {
    All,                 // every active peer
    Some(Vec<P>),        // specific list
    One(P),              // a single peer
}
```

When you call `sender.send(Recipients::All, message, priority)`:

1. Look up the current active peer set.
2. For each peer, check rate limit. If at limit, skip.
3. For each peer not rate-limited, send.
4. Return the list of recipients we attempted to send to.

The `LimitedSender::check` returns a `CheckedSender` if any peer is
sendable, or an error with `SystemTime` (when the earliest rate-limited
peer becomes available) if all are rate-limited.

## Appendix G — Rate limiting deep dive

`p2p` uses the `governor` crate for rate limiting. Each peer has a
`Quota` (e.g., `Quota::per_second(NonZeroU32!(10))` = 10 messages per
second per peer per channel).

The `RateLimiter` tracks per-peer usage. When a peer hits its quota,
subsequent messages to that peer are rejected until the quota refills.

**Why per-peer AND per-channel?** Without per-channel limits, a peer
could fill one channel's quota and starve others. Per-channel limits
ensure fair sharing of bandwidth.

**Default values** (`config.rs:129-194`):

- Local: 1 concurrent handshake per IP, 5s timeout.
- Single-region: 16 concurrent handshakes per IP, 5s timeout.
- Multi-region: 128 concurrent handshakes per IP, 10s timeout.

These are conservative defaults for production. Tune up for high-churn
networks, down for constrained environments.

## Appendix H — The `block!` macro in detail

`p2p/src/lib.rs:391-398`:

```rust
macro_rules! block {
    ($blocker:expr, $peer:expr, $($arg:tt)+) => {
        let peer = $peer;
        tracing::warn!(peer = ?peer, $($arg)+);
        #[allow(clippy::disallowed_methods)]
        $blocker.block(peer)
    };
}
```

Expands to:

```rust
let peer = $peer;
tracing::warn!(peer = ?peer, ...);  // structured log
blocker.block(peer)                // sever connection + future-refuse
```

The `peer = ?peer` syntax in `tracing` makes the peer's public key a
**field** of the log entry. You can filter logs by peer in your log
aggregator.

The macro is the **only sanctioned way** to block a peer. Direct calls
to `blocker.block()` are discouraged (and lint-flagged via
`clippy::disallowed_methods`).

## Appendix I — Message size and bandwidth accounting

The `Config` exposes `max_message_size` (default ~1 MB). Each message
must fit within this limit, or it gets dropped.

The actual on-wire size is larger (handshake overhead, codec overhead,
signatures). The simulator (chapter 02) accounts for this in its
bandwidth model — every message contributes to the sender's egress and
receiver's ingress in the bandwidth simulation.

For a 1 MB message on a 10 MB/s egress link:
- Transmission time = 1 second (transmission delay).
- Plus latency (e.g., 10 ms).
- Plus jitter (random ± 3 ms).
- = ~1.01 seconds total.

The simulator's bandwidth model (chapter 02's progressive filling) is
what makes these numbers realistic.

## Appendix J — The `Connection`-level state machine vs channel-level

Each TCP connection (the bottom layer) is **independent** of each
logical channel (the top layer). One connection carries many channels.

```
+----------------+   +----------------+   +----------------+
|  Channel 0     |   |  Channel 1     |   |  Channel 2     |
|  (votes)       |   |  (certs)       |   |  (resolver)    |
+--------+-------+   +--------+-------+   +--------+-------+
         |                    |                    |
         +--------------------+--------------------+
                              |
                   +----------+----------+
                   |  Connection          |
                   |  (X25519 + ChaCha20) |
                   |  per-peer             |
                   +---------------------+
```

The same physical connection carries all registered channels. Each
channel has its own mailbox (chapter 15), its own rate limit, its own
receiver task.

If a channel's mailbox is full, the connection keeps working. Other
channels continue.

## If you only remember three things

1. **Every message has a `channel` ID.** It's how the network routes Data to your handlers. Register channels at startup with quotas.

3. **The `Blocker` trait + `block!` macro = your slashing layer.** When a peer misbehaves, block + log. Structured, automatic, mandatory.
